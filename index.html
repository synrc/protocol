• Sokhatsky Messenger Protocol

Protocol

Sokhatsky Messenger Protocol: A three-layered cryptographic communication suite for serverless, brokered, and document-centric messaging.

Abstract

Sokhatsky Messenger Protocol (SMP) is a comprehensive communication framework designed for high-security environments. It spans from low-level serverless UDP multicast to high-level document-centric ERP/1 systems, ensuring reliability through formal ASN.1 specifications and modern cryptographic foundations. SMP is built to satisfy both civilian and military communication requirements, incorporating X.509 PKI and X.400 MHS/IPMS patterns.

MESSAGE X.894

CMS secured UDP multicast serverless communication for local area networks.

V1: Serverless Multicast

This layer MESSAGE-v1.asn1 defines the core primitives for peer-to-peer and multi-user communication without the need for a central broker. It uses UDP multicast for discovery and delivery, featuring authority registration and file sharing.

MESSAGE-v1 DEFINITIONS ::= BEGIN

    IMPORTS ContentInfo FROM CryptographicMessageSyntax-2009
            Certificate FROM PKIX1Explicit-2009 ;

    OS  ::= ENUMERATED { apple (1), microsoft (2), google (3),
                         ericsson (4), nokia (5), cisco (6) }

    P2P ::= SEQUENCE { from OCTET STRING, to OCTET STRING }
    MUC ::= SEQUENCE { room OCTET STRING }

    CHATString ::= OCTET STRING

    -- 01 Authority (CA)

    Feature ::= SEQUENCE {
       id OCTET STRING,
       key OCTET STRING,
       value OCTET STRING,
       group OCTET STRING }

    AuthorityMethod ::= ENUMERATED { announce (0), register (1),
       authenticate (2), dismiss (3), renew (4) }
    Authority ::= SEQUENCE {
       session OCTET STRING,
       type AuthorityMethod,
       cert OCTET STRING,
       settings SEQUENCE OF feature Feature }

    -- 02 Message (CHAT)

    File ::= SEQUENCE {
       id OCTET STRING,
       mime OCTET STRING,
       payload ANY,
       parentid OCTET STRING }

    Channel ::= SEQUENCE { channel OCTET STRING }
    MessageType ::= ENUMERATED { sys (1), reply (2), forward (3), read (4), edited (5) }
    MessageStatus ::= ENUMERATED { async (1), delete (2), clear (3), update (4), edit (5) }
    MessageFeed ::= CHOICE { p2p [0] P2P, muc [1] MUC, chan [2] Channel }
    Message ::= SEQUENCE {
       id CHATString,
       feed MessageFeed,
       from CHATString,
       to CHATString,
       files SEQUENCE OF file File,
       type MessageType,
       link CHOICE { empty [1] NULL, message [2] Message },
       seenby CHATString,
       repliedby CHATString,
       mentioned SEQUENCE OF name CHATString,
       status MessageStatus DEFAULT clear }

    Ack ::= SEQUENCE { table OCTET STRING, id OCTET STRING }

    -- 05 Activity (Presense, Typing, Async Messages)

    Activity ::= SEQUENCE {
       id OCTET STRING,
       vsn Version,
       session OCTET STRING,
       nickname OCTET STRING,
       comments SEQUENCE OF comment OCTET STRING }

CHAT X.509

TLS unicast broker and PKI X.509 infrastructure (ПК КЗІ X.509).

V2: Brokered TLS & PKI

The second layer CHAT-v2.asn1 introduces a central broker architecture with TLS encryption. It implements full X.509 PKI support, LDAP-based search, and complex subscription/presence mechanisms for global messaging systems.

00 Core Types

Version ::= INTEGER { v0(0), v1(1), v2(2), v3(3) }
OS ::= ENUMERATED { apple(1), microsoft(2), google(3),
                    ericsson(4), nokia(5), cisco(6) }
  • Version: Integer representing the protocol version, from legacy v0 to current v3.
  • OS: Enumeration of supported operating systems for push notifications and device-specific optimizations.

01 Authority

Feature ::= SEQUENCE {
   id OCTET STRING,
   key OCTET STRING,
   value OCTET STRING,
   group OCTET STRING }
  • id: Globally unique identifier for the configuration feature.
  • key: Semantic name of the feature or capability being negotiated.
  • value: Current operational value or parameter for the specified key.
  • group: Categorical classification to isolate features by domain.
AuthorityMethod ::= ENUMERATED { announce(0), enroll(1),
   authenticate(2), revoke(3), renew(4), ... }
  • announce: Protocol primitive for authority presence discovery on the network.
  • enroll: Entry-point for new entities to request domain inclusion (PKI enrollment).
  • authenticate: Verification of entity identity through cryptographic challenges.
  • revoke: Administrative decommissioning of an entity's privileges or credentials.
  • renew: Extension of validity for existing certificates or session authorities.
Authority ::= SEQUENCE {
   id OCTET STRING,
   vsn Version,
   session OCTET STRING,
   type AuthorityMethod,
   cert OCTET STRING,
   settings SEQUENCE OF feature Feature, ... }
  • id: Stable identifier of the administrative authority within the protocol.
  • vsn: Protocol version currently enforced by the authority.
  • session: Contextual session token for stateful authority interactions.
  • type: The specific administrative action defined by the AuthorityMethod.
  • cert: Binary encoding of the X.509 certificate or public key material.
  • settings: A collection of active features governing the entity's behavior.

02 Message

MessageType ::= ENUMERATED { normal(0), sys(1),
   reply(2), forward(3), ... }
  • normal: Standard user-to-user interpersonal communication.
  • sys: Automated system notification or administrative control message.
  • reply: A message tied to a previous context through threading.
  • forward: Content relayed from another source, preserving original context.
MessageFeed ::= CHOICE { private [1] Privatebox,
   chan [2] Streambox, mailbox [4] Mailbox,
   group [8] Groupbox }
  • private: Encapsulates information for single-user private delivery.
  • chan: Targeting a specific broadcast or multicast stream.
  • mailbox: Document-oriented priority delivery system.
  • group: Context for multi-user conference interaction.
Message ::= SEQUENCE {
   id OCTET STRING,
   vsn Version,
   session OCTET STRING,
   feed MessageFeed,
   payload ContentInfo,
   type MessageType OPTIONAL,
   linkId OCTET STRING OPTIONAL,
   files SEQUENCE OF fileId OCTET STRING OPTIONAL,
   mentioned SEQUENCE OF name OCTET STRING OPTIONAL, ... }
  • id: Irreversible unique identifier for message tracking and deduplication.
  • vsn: Versioning of the message schema for forward compatibility.
  • session: Transient session identifier linking the message to a transport context.
  • feed: Dispatch routing information determining the message destination.
  • payload: Cryptographically secured content (CMS) of the message.
  • type: Semantic classification of the message for UI and processing logic.
  • linkId: Reference to parent message ID for threading and replies.
  • files: Collection of unique identifiers for binary attachments.
  • mentioned: List of entity identifiers tagged for special notification.

03 Inbox

InboxOp ::= ENUMERATED { get(1), erase(2), ... }
  • get: Protocol constant for inbox retrieval requests.
  • erase: Protocol constant for inbox cleanup requests.
InboxQueryRequest ::= SEQUENCE {
   op InboxOp,
   feed MessageFeed OPTIONAL, ... }
  • op: Selection of the specific inbox operation to execute.
  • feed: Optional filter to scope the query to a specific communication feed.
InboxQueryResult ::= SEQUENCE {
   items SEQUENCE OF item InboxItem OPTIONAL, ... }
  • items: Aggregated list of inbox results matching the query criteria.

04 Event

MessageEventType ::= ENUMERATED { received(1),
   delivered(2), read(3), edited(4), deleted(5),
   updated(6), ... }
  • received: Signal that the transport has accepted the message.
  • delivered: Confirmation that the message reached the target device.
  • read: Indication that the recipient viewed the content.
  • edited: Notification that original content was modified.
  • deleted: Signal to remove message from local storage.
MessageEvent ::= SEQUENCE {
   type MessageEventType,
   messageId OCTET STRING OPTIONAL,
   feed MessageFeed OPTIONAL,
   actor OCTET STRING OPTIONAL,
   timestamp INTEGER OPTIONAL, ... }
  • type: Specific message lifecycle event being reported.
  • messageId: Identifier of the message affected by this event.
  • feed: Communication channel where the message originates.
  • actor: Identification of the entity that triggered the event.
  • timestamp: Precise UTC time when the event occurred.
PresenceEventType ::= ENUMERATED { online(1),
   offline(2), typing(3), ... }
  • online: The entity has become available for interaction.
  • offline: The entity is no longer reachable.
  • typing: Activity indicator for content composition.
PresenceEvent ::= SEQUENCE {
   type PresenceEventType,
   feed MessageFeed OPTIONAL,
   actor OCTET STRING OPTIONAL,
   timestamp INTEGER OPTIONAL, ... }
  • type: Specific presence transition being reported.
  • feed: Optional channel context for the presence update.
  • actor: The entity whose presence state has changed.
  • timestamp: Time of the presence update for synchronization.
Event ::= SEQUENCE {
   id OCTET STRING,
   vsn Version,
   session OCTET STRING,
   payload EventPayload, ... }
  • id: Unique event sequence identifier.
  • vsn: Protocol version for event encoding.
  • session: Network session context for event delivery.
  • payload: Actual event content (Message or Presence).

05 Query Layer

QueryType ::= ENUMERATED { get(1), set(2),
   result(3), error(4), ... }
  • get: Request for personal or global state from the server.
  • set: Directive to modify or create state on the server.
  • result: Response containing requested data or confirmation.
  • error: Report of operation failure or protocol violation.
Query ::= SEQUENCE {
   id OCTET STRING,
   vsn Version,
   session OCTET STRING,
   queryType QueryType,
   refId OCTET STRING OPTIONAL,
   feed MessageFeed OPTIONAL,
   target OCTET STRING OPTIONAL,
   payload QueryPayload OPTIONAL,
   code INTEGER OPTIONAL, ... }
  • id: Unique correlation identifier for tracking.
  • vsn: Protocol schema version for the query.
  • session: Active session token for query authorization.
  • queryType: High-level intent of the interaction (CRUD).
  • refId: Correlation ID for linking responses to requests.
  • feed: Contextual channel targeting for the query.
  • target: Routing hint for specialized service domains.
  • payload: Family-specific envelope containing typed data.
  • code: Numerical return code for status reporting.

06 Search

SearchQueryRequest ::= SEQUENCE {
   op SearchOp,
   dn OCTET STRING OPTIONAL,
   field OCTET STRING OPTIONAL,
   value OCTET STRING OPTIONAL,
   criteria Criteria OPTIONAL,
   type Scope OPTIONAL, ... }
  • op: Specific search operation to be performed.
  • dn: Distinguished Name base path for scoped searches.
  • field: Attribute or property targeted by the search.
  • value: Search term or pattern to match against.
  • criteria: Logic for matching (equality, pattern, negation).

12 Protocol

CHATProtocol ::= CHOICE {
   authority [1] Authority,
   message [2] Message,
   event [4] Event,
   query [5] Query }
  • authority: Administrative and PKI-related interactions.
  • message: Core communication and data exchange events.
  • event: Real-time lifecycle and presence synchronization.
  • query: Structural request/response layer for state management.
CHATMessage ::= SEQUENCE {
   no INTEGER,
   headers SEQUENCE OF header OCTET STRING,
   body CHATProtocol }
  • no: Sequence number for reliable transport and ordering.
  • headers: Metadata headers for routing and security.
  • body: Actual protocol payload (Authority/Message/Event/Query).

MAIL X.420

ERP/1: Documents protocol of Order №40 MoD implemented as MHS IPMS Priority Mail.

V3: Document-Centric MHS/IPMS

The third layer MAIL-v3.asn1 is an extension of v2, incorporating InterPersonal Messaging Service (IPMS) and Message Handling System (MHS) patterns from X.400. It is tailored for official document flow, implementing strict ERP/1 requirements and the Ministry of Defense Order №40.

MAIL-v3 DEFINITIONS IMPLICIT TAGS ::= BEGIN

    IMPORTS CHATProtocol, Message, File FROM CHAT-v2 ;

    --- IPMS/MHS Extension for ERP/1

    IPMSHeading ::= SEQUENCE {
        originator      OCTET STRING,
        authorizing     SEQUENCE OF OCTET STRING OPTIONAL,
        primaryRecipients SEQUENCE OF OCTET STRING,
        copyRecipients    SEQUENCE OF OCTET STRING OPTIONAL,
        blindCopyRecipients SEQUENCE OF OCTET STRING OPTIONAL,
        subject         OCTET STRING,
        priority        INTEGER,
        expiryTime      GeneralizedTime OPTIONAL,
        replyTime       GeneralizedTime OPTIONAL,
        importance      ENUMERATED { low(0), normal(1), high(2) },
        sensitivity     ENUMERATED { personal(1), private(2), companyConfidential(3) }
    }

    DocumentType ::= ENUMERATED { draft(0), official(1), signed(2), archived(3) }

    Order40Metadata ::= SEQUENCE {
        documentId      OCTET STRING,
        controlDate     GeneralizedTime,
        executor        OCTET STRING,
        signature       OCTET STRING,
        regulatoryAct   OCTET STRING -- Order №40 MO
    }

    MailMessage ::= SEQUENCE {
        heading IPMSHeading,
        body    SEQUENCE OF CHOICE { 
            text [0] OCTET STRING, 
            doc  [1] DocumentContent 
        },
        meta    Order40Metadata 
    }

    DocumentContent ::= SEQUENCE {
        vsn     INTEGER,
        type    DocumentType,
        content ANY,
        files   SEQUENCE OF File
    }

END

SMP Testing DSL

Domain Specific Language for testing and scripting SMP interactions.

DSL Session

The console testing DSL allows for rapid prototyping and validation of protocol flows, from simple P2P messages to complex document submissions.

# SMP Testing Session
connect user@domain.com
auth password "secret"
login cert "path/to/cert.der"

# Presence
set presence online "Available"
send presence to alice subscribed

# Messages
send message to alice "Hello there!"
send message to room:general "Hi everyone" reply-to "msg-id-123"

# Rich message
send message {
  to: alice
  subject: "ERP/1 Document Draft"
  body: "Please review the attached Order 40 document."
  priority: high
}

# Acknowledgements
send ack for "msg-id-123"
send read for "msg-id-xyz"

# Group Chat
join room general
invite user bob to room general

# File Transfers
send file "order_40.pdf" to alice
request file "document-id" from bob

# Low level/Raw
send raw hex "3082..."
expect message from alice matching ".*received.*"
